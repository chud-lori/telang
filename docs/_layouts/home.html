Self-host object storage

Use a private Telegram channel
as your disk.

Telang is a single Go binary you run on your own box. It speaks the standard S3 wire protocol on the front, so anything that talks to S3 (aws s3, rclone, boto3, Cyberduck, aws-sdk-go-v2, …) Just Works. The bytes live as encrypted messages inside Telegram.

Get started → View on GitHub

Hobby projects, internal tools, dev / staging. Not for production, customer data, or high-throughput public assets — Telegram can ban the account and your bytes are gone.

What you get

Drop-in S3 API

SigV4 headers and presigned URLs, ListObjectsV2 with prefix/delimiter/pagination, multipart upload, and HTTP Range — all wired up.

AES-256-GCM end to end

Every object is encrypted in 64 KB authenticated frames before it leaves the box. Telegram only ever sees ciphertext.

Two storage modes

bot — 20 MB per object, no phone number, simplest. mtproto — 2 GB per object via a user-account session.

LRU ciphertext cache

Disk-backed cache so warm reads stay on local disk and never round-trip Telegram. Configurable size, evictions on access.

Minimal browser UI

Server-rendered listing, download links, upload form, password-gated delete. No JS build step.

Operator tools

telang init, reauth, fsck, export-metadata, import-metadata — the boring stuff is already in the binary.

Three commands to running daemon

  1. 1 Install the binary. One file, no runtime deps.
  2. 2 Run telang init. Paste your Telegram credentials, get S3 keys back.
  3. 3 Start telang serve. Point your S3 client at http://localhost:9000.
{% highlight bash %} go install github.com/{{ site.github.repository_nwo | default: 'chud-lori/telang' }}/cmd/telang@latest telang init \ --config ./config.toml \ --keys ./keys.toml \ --data-dir ./var telang serve --config ./config.toml {% endhighlight %}

Talk to it from your code

Any S3 SDK works — there is no Telang-specific client library. Point the official AWS SDK at http://localhost:9000 and use the credentials telang init printed.

Go aws-sdk-go-v2
{% highlight go %} ctx := context.Background() cfg, _ := config.LoadDefaultConfig(ctx, config.WithRegion("tg-1"), config.WithCredentialsProvider(credentials.NewStaticCredentialsProvider( "AKIA...", "...", "", )), config.WithBaseEndpoint("http://localhost:9000"), ) c := s3.NewFromConfig(cfg, func(o *s3.Options) { o.UsePathStyle = true }) _, _ = c.CreateBucket(ctx, &s3.CreateBucketInput{ Bucket: aws.String("photos"), }) _, _ = c.PutObject(ctx, &s3.PutObjectInput{ Bucket: aws.String("photos"), Key: aws.String("holiday.jpg"), Body: bytes.NewReader(data), }) {% endhighlight %}
Python boto3
{% highlight python %} import boto3 from botocore.config import Config s3 = boto3.client( "s3", endpoint_url="http://localhost:9000", aws_access_key_id="AKIA...", aws_secret_access_key="...", region_name="tg-1", config=Config( signature_version="s3v4", s3={"addressing_style": "path"}, ), ) s3.create_bucket(Bucket="photos") with open("holiday.jpg", "rb") as f: s3.put_object(Bucket="photos", Key="holiday.jpg", Body=f) {% endhighlight %}

Be honest with yourself

Telang is a small tool with sharp edges. Read this before you bet on it.

Account ban

Telegram tolerates this kind of usage at small scale but doesn't promise to. A ban means every object is unreadable until you import the metadata into a fresh account and re-fetch.

keys.toml loss

No keys, no plaintext. Telang refuses to start if the file isn't chmod 600. Back it up off-host the moment you create a new bucket.

Throughput

Telegram rate-limits aggressive callers. Telang honours FLOOD_WAIT and backs off, so a sudden spike is silently slowed, not failed. Hobby workloads only.
